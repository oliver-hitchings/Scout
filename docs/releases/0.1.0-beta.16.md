# Scout 0.1.0-beta.16

Beta 16 adds a protected recovery path for a VPS that was provisioned against the wrong private workspace.

## Changes

- Adds a host-local-only operation that adopts an existing private Scout repository without exposing workspace contents to the release deployment account.
- Clones and validates the private workspace before activation, restores existing encrypted recovery data (or creates it for a legacy repository), completes a backup push, and atomically swaps the configured workspace.
- Retains the previous VPS workspace beside the active one as a timestamped rollback copy.
- Adds a protected GitHub Actions maintenance workflow for preparing a repository-scoped deploy key and verifying adoption.
- Verifies the expected CV source count, renders every tailored CV with managed Typst, and requires automatic backup to report `synced`.
- Corrects the VPS CV health counter to accept the current boolean availability fields.
- Requires encrypted private backup before remote clients can mutate canonical VPS state, while keeping recovery administration host-local.
- Documents a reusable VPS, Tailscale, encrypted-backup, recovery-drill, and multi-client acceptance procedure.
- Adds a host-local recovery-passphrase rotation that preserves the existing data key, encrypted blobs, and emergency recovery key for an already-unlocked canonical VPS.

## Recovery key

When encryption is first enabled on the surviving local workspace, save and confirm the one-time emergency recovery key. The VPS reuses that existing encrypted backup and the same passphrase; it does not rotate the recovery key. The recovery passphrase or emergency key can restore encrypted credentials, generated documents, and chat transcripts on a replacement device.
