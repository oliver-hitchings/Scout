# Scout 0.1.0-beta.19

Beta 19 fixes the dead ends a new install could hit during first-run setup, lets each scheduled scan choose which days it runs, improves which jobs a scan finds and whether they still exist, and brings model choice and provider headroom to the conversation engine picker.

## First-run setup and dialogs

- Fixes a script error on a brand-new install. The uninitialised opportunity response carried an incomplete shape, which stopped the dashboard rendering behind the setup dialog. Both the empty and populated responses now derive from one definition so they cannot diverge again.
- Confirms workspace creation immediately instead of leaving the welcome screen unchanged under a frozen progress message, and prevents a second submission that reported the workspace already existed.
- Stops an established workspace being held in first-run onboarding. A staged but unactivated proposal, or a workspace whose readiness lapsed, previously reopened the blocking wizard on every load with no close control.
- Opens the settings section that each retune control names. "Retune my search" reached the AI provider question and "Retune search and sources" reached the name question.
- Makes controls that looked like choices behave like them: the backup panel's "Not now" and the first-run restore form now open and close the panel they belong to.
- Corrects the provider step message that reported a sign-in problem when a provider was already signed in and simply unselected.

## Scheduled scans

- Each scan job now chooses the days it runs. Presets cover every day, weekdays only, custom days, and alternating with the other provider, which gives one job Sunday, Monday, Wednesday and Friday and the other Tuesday, Thursday and Saturday so two providers never scan on the same day.
- Windows uses a weekly task trigger for a day subset, Linux restricts `OnCalendar` to the selected days in the workspace timezone, and macOS lists one calendar entry per weekday. A job set to every day keeps its simpler daily trigger.
- `scout schedule install` accepts `--days 0,1,3,5`, counting Sunday as `0`.

## Scan results

- Every configured source now reaches assessment. Candidates were previously collected in source order until a fixed cap, so once the cap filled, later sources contributed nothing and the loss was not recorded. Each source receives a guaranteed share, the remainder is shared, and anything that did not fit is reported in the scan record.
- Adverts are checked before scoring, so closed roles no longer reach reports. Only clear evidence closes an advert; timeouts, blocks and server errors leave a candidate in place, so an offline host can never close a tracker in bulk.
- A verification pass now re-examines what the day's primary scan decided rather than repeating discovery, which substantially reduces the cost of running a second provider.
- The assessment request carries only the fields it uses and the scoring-relevant configuration, and hard exclusions are applied before the request using the same rule applied after it.

## Conversations

- Choosing an engine now shows how much of each provider's allowance is used, with the reset time where the provider reports one, and offers a model for that conversation.
- The chosen model is saved with the job's conversation and restored when it is reopened. It is fixed once the conversation has a provider session, because a resumed session is bound to the model it started with.
- Model spend for Claude is shown per model where the local records allow it, and is labelled as spend. Remaining allowance is only available per provider, and is not presented per model.
- Fixes the usage summary, which labelled the first provider limit window as five hours. An account whose first window is weekly reported a spent weekly allowance as a spent five-hour one.

## Upgrade and compatibility

- No workspace schema migration, provider sign-in, or backup-format change is required.
- Schedules saved before this release have no day selection and continue to run every day.
- Conversations saved before this release have no model and continue to follow the provider default configured in settings.
- Release tags are immutable. A published tag is never moved; a correction is published as a new version.
